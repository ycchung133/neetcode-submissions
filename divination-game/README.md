# 演算法籤詩（Divination Game）

把 NeetCode 150 包成**廟宇擲筊解籤**風格的遊戲，重用你既有的兩款 Android 遊戲資產：

- **擲茭** 的剛體拋擲物理 → 落地判定 3 種狀態
- **吊酒瓶** 的繩索/鉤子/約束/平衡物理
- **解籤 LLM** → 遊戲裡的「AI 解籤師／導師」

## 為什麼契合

| 既有資產 | 核心技術 | 在本遊戲的角色 |
|---|---|---|
| 擲茭 | 拋擲剛體 → 離散落地狀態（聖/笑/陰筊） | 「物理表現 → 離散狀態機判定」的現成範例；用於決策/commit、隨機結構、Heap 呈現 |
| 吊酒瓶 | 繩索 + 鉤子 + 擺盪 + 約束 + 平衡 | Linked List（繩連節點）、Tree（吊飾）、BST 平衡、精準約束解謎 |
| 解籤 LLM | 籤號→籤詩→情境化詮釋 | AI 解籤師＝導師：把玩家的解題嘗試「當籤來解」，給漸進式提示 |

## 設計原則：引擎與 LLM 都是「可插拔介面」

我們**不綁定**特定引擎或平台。核心玩法邏輯只依賴 `contracts/` 裡的抽象契約；
既有的擲茭/吊酒瓶引擎與解籤 LLM 各寫一個 adapter 去實作它。
→ 之後接 Unity、Android 原生或 Web，都只換 adapter，不動玩法邏輯。

```
遊戲玩法邏輯（平台無關）
        │ 依賴
        ▼
contracts/  ← 抽象契約（物理引擎、解籤 LLM）
        ▲ 實作
        │
  ┌─────┴─────┬──────────────┐
Unity adapter  Android adapter  Web adapter(matter.js)
（重用擲茭/吊酒瓶）              （../game 的 PoC）
```

## 目前狀態
- `contracts/` — 平台無關的介面契約（物理引擎 + 解籤 LLM），先用 TypeScript 型別表達，等同規格文件。
- `DESIGN.md` — 籤詩主題的關卡與敘事設計。
- 引擎 adapter：**待你提供既有專案原始碼後填入**（見下）。

## 要真正重用既有引擎碼，先把它接進來
本 session 跑在雲端容器，讀不到你電腦的檔案。三選一：
1. **推到 GitHub**（私有也行）→ 我用 `add_repo` 加入後直接讀碼重用（推薦）。
2. **貼關鍵檔**（引擎入口、物理參數、解籤 prompt）到對話，我照著移植。
3. **改用本機版 Claude Code**（CLI/桌面），它能直接讀你電腦裡的擲茭/吊酒瓶專案。
