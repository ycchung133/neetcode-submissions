# 《AlgoPhysics 150》遊戲開發計畫書

> 主題：NeetCode 150 × 物理模擬 × LLM
> 一句話定位：用「物理沙盒解謎 + AI 導師」把 NeetCode 150 的資料結構與演算法變成可以動手玩的關卡。

---

## 一、專案概述

| 項目 | 內容 |
|---|---|
| 專案代號 | **AlgoPhysics 150**（暫定中文名：演算之境） |
| 定位 | 物理化的 DSA 學習遊戲，結合即時 AI 導師 |
| 目標平台 | Web 優先（桌面瀏覽器），未來延伸 Steam / 行動裝置 |
| 目標客群 | 準備技術面試的學生與工程師、想學 DSA 但覺得刷題枯燥的人 |
| 核心賣點 | ①演算法「看得見、摸得到」的物理化呈現 ②LLM 作為即時導師、關卡出題者與對話 NPC |

**核心理念**：把每個資料結構變成物理實體（球、彈簧、繩、方塊），把每個演算法變成一段物理過程；玩家透過操控物理世界來「執行」演算法，由 LLM 導師即時解釋動作對應到的程式碼邏輯。

---

## 二、遊戲核心概念

### 2.1 世界觀
玩家是一位演算法工程師，被吸入由資料構成的異世界「Data Realm」，該世界的物理法則即資料結構規則。要離開，必須修復 150 座對應 NeetCode 150 的「演算法神殿」。

### 2.2 三大支柱
1. **物理化的資料結構**：陣列＝軌道方塊、鏈結串列＝繩連浮球、二元樹＝重力吊飾、堆積＝浮力水槽。
2. **可操作的演算法**：用物理互動完成演算法（推指針球、剪重繩子）。
3. **LLM 智慧層**：即時導師、動態提示、自然語言下指令、關卡敘事與 boss 對話。

---

## 三、玩法設計

### 3.1 關卡對應（依 NeetCode 150 分類）

| NeetCode 主題 | 物理隱喻 | 玩法範例 |
|---|---|---|
| Arrays & Hashing | 軌道方塊 / 記憶魔法陣 | Two Sum：射出互相吸引的球湊出目標和才湮滅 |
| Two Pointers | 左右磁力指針球 | Container With Most Water：調柱子讓水位淹沒最大面積 |
| Sliding Window | 可伸縮力場框 | 拖動邊界框住符合條件的區間 |
| Stack | 重力堆疊塔 | Valid Parentheses：括號落下必須配對消除否則塔倒 |
| Linked List | 繩連浮球 | Reverse List：實際剪斷、翻轉、重接繩子 |
| Trees | 重力吊飾 | Invert Tree：左右鏡射整棵吊飾並維持平衡 |
| Heap / PQ | 浮力水槽 | 最大值因浮力上浮，取出頂端 |
| Backtracking | 分岔光路迷宮 | Subsets：每節點選走或不走，回溯時光路收回 |
| Graphs | 節點彈簧網 | BFS/DFS：能量波以物理方式在彈簧網擴散 |
| Binary Search | 天平二分 | 每次把搜尋空間物理對半切開 |

### 3.2 三種難度模式
- **Sandbox**：純物理互動、理解直覺、無時間壓力。
- **Challenge**：限定操作步數 ＝ 時間複雜度，逼玩家找最優解。
- **Code Duel**：玩家實際寫程式碼，遊戲即時把 code 視覺化成物理動畫執行，錯就物理崩塌。← 銜接真正刷題。

### 3.3 進度系統
- 150 題 → 150 座神殿，分 9 大區逐區解鎖。
- 每題三星：完成 / 最優複雜度 / 不使用提示。
- 星星解鎖進階講解與新物理道具。

## 四、物理模擬設計

### 4.1 技術選型
- 引擎：**matter.js**（2D、Web 友善、輕量）為主；3D 吊飾/繩索評估 **Rapier.js**（WASM、效能佳、支援關節與軟體約束）。
- 2D 優先：資料結構本質是拓樸關係，2D 已能清楚表達，開發成本低、效能穩。

### 4.2 需模擬的物理元素
| 元素 | 物理特性 | 對應資料結構 |
|---|---|---|
| 剛體方塊 | 碰撞、堆疊、重力 | Array / Stack |
| 繩索 / 關節 | 張力、可剪斷可連接 | Linked List / Tree edges |
| 彈簧網 | 力傳導、震盪 | Graph |
| 浮力場 | 密度＝數值大小，大者上浮 | Heap |
| 磁力 | 指針球吸引或排斥 | Two Pointers |
| 力場框 | 可縮放觸發區 | Sliding Window |

### 4.3 關鍵技術挑戰
- **物理與邏輯一致性**：物理層負責「表現」，另設離散狀態機負責「正確性判定」，用事件同步（剪繩 → 觸發 `unlink(node)` 邏輯事件）。
- **可重現性**：固定 timestep + 種子，確保關卡可複現、可自動測試。

## 五、LLM 應用設計（核心差異化）

採用 **Claude API**（Opus / Haiku 分層），四個子系統：

### 5.1 智慧導師 Tutor
- 把「當前物理世界狀態 + 操作歷史」序列化餵給 LLM，給漸進式提示（概念 → 方向 → 接近解法），不直接給答案。
- 用 tool use 讓 LLM 能讀取關卡狀態、在畫面上高亮元件，邊講解邊指。

### 5.2 動態出題官 Generator
- 在既有題框上生成變體題與測資，讓 150 題可無限延伸，解決背答案問題。
- 生成後由確定性驗證器檢查合法性（LLM 出題、程式驗題，避免幻覺）。

### 5.3 自然語言指令 NL-Command
- 玩家打字/語音下指令 → LLM 解析成遊戲 API 呼叫 → 執行。把「意圖」翻譯成「演算法步驟」。

### 5.4 敘事與 Boss 對話 Narrative
- 每區守關 Boss 是演算法擬人化角色（如「暴力破解魔王 O(n²)」），LLM 驅動即時對話，玩家用「更優解論述」擊敗它。

### 5.5 LLM 工程重點
- **成本控制**：提示用 Haiku、深度講解/出題用 Opus；快取常見回應。
- **防作弊 / 防提示注入**：玩家輸入視為不可信，系統 prompt 與遊戲狀態分離，工具呼叫加白名單。
- **降低延遲**：串流輸出、預生成提示、本地規則先擋簡單情況。

## 六、技術架構

```
前端 (Web)
  - React + TypeScript (UI / 關卡選單)
  - Canvas / PixiJS (渲染)
  - matter.js / Rapier.js (物理)
  - 離散狀態機 (演算法正確性判定)
  - Monaco Editor (Code Duel 寫程式)
        │
後端 (Node.js / serverless)
  - LLM Gateway (呼叫 Claude API、快取、限流)
  - 題目/測資驗證器 (沙箱執行玩家程式碼)
  - 進度存檔 / 帳號
```

- 前端優先、後端輕量：物理與渲染全在瀏覽器；後端只做 LLM 代理、程式碼沙箱與存檔。
- 資料來源：沿用本 repo 的 NeetCode 解題作為 Code Duel 參考解與視覺化素材。

| 階段 | 期程 | 產出 |

## 九、風險與對策

| 風險 | 影響 | 對策 |
|---|---|---|
| 物理與演算法邏輯不一致 | 高 | 物理管表現、狀態機管判定，事件同步；PoC 先驗證 |
| LLM 幻覺/給錯提示 | 高 | 提示走概念→方向不給答案；出題由程式驗證 |
| LLM 成本失控 | 中 | 分層模型、快取、本地規則先擋 |
| 150 題內容量龐大 | 中 | 用物理元件模板讓題目可組裝複用 |
| 覺得像玩具不像刷題 | 中 | Code Duel 模式銜接真實寫程式 |
| 物理引擎跨瀏覽器效能 | 低 | 固定 timestep、2D 優先、WASM 備援 |

## 十、成功指標（KPI）

- 學習成效：玩家在對應 NeetCode 真題正確率提升 ≥ 30%。
- 留存：次日 ≥ 40%，7 日 ≥ 20%。
- 參與：平均每題 LLM 導師互動 ≥ 2 次、滿意度 ≥ 4/5。
- 內容：MVP 涵蓋 9 大主題、≥ 80 題可玩。
